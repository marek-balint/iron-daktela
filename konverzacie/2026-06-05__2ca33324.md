Added a **two-tier model routing** strategy to `docs/04-ai-classification.md`, plus env vars in doc 06 and a note in `CLAUDE.md`.

## How it works

- **Tier 1 — cheap triage (every ticket):** a fast/cheap model classifies all incoming tickets — simple product questions ("in stock?", "what colour?"), FAQs, spam detection. Candidate: **Gemini 1.5 Flash** (your suggestion), or Claude Haiku. It also returns a `needs_deep_reasoning` / `complexity` flag.
- **Tier 2 — strong model (Anthropic Claude):** escalate only when it's worth it — **order issues** (where's my order, refunds, payment, complaints), **big/multi-part product questions**, low confidence, or already high-priority tickets.

Routing rules are a config table (category + complexity → tier), so you can rebalance cost vs quality without code changes. The **output schema stays identical** across tiers, so the priority scorer doesn't care which model ran — and a `model_used` field is stored for auditing/cost tracking.

## One decision to flag

Mixing providers (Gemini for Tier 1 + Claude for Tier 2) means a **second SDK + `GEMINI_API_KEY`** and slightly different JSON-mode behavior. If that overhead isn't worth it, **Claude Haiku (Tier 1) + Claude Sonnet (Tier 2)** gives the same cheap/strong split on a single provider/SDK. I documented both paths in the env file — you can decide when we get to implementation.

One small heads-up: it's June 2026 and Gemini 1.5 Flash is fairly old now. If you go mixed-provider, you may want a newer cheap Gemini tier — but the routing design is model-agnostic, so swapping the Tier 1 model later is just a config change.
